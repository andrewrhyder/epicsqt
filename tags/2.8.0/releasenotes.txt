SUMMARY:
QEframework customisation:
==========================
The QE framework has always facilitated custom application development, but has attempted to reduce the need for custom applications by providing a rich presentation application (QEGui) and a wide range of QE widgets that allow sophisicated requirements to be fully satisfied by generic QE framework components. This release includes enhancements to support this philosophy as follows:
- QE widget / application interaction.
    Applications can easily interact with named widgets in a dynamicly loaded GUI. An application can load a .ui file, then make a request to the QE framework that one of its tool bar buttons be associated with a particular function of a named QE widget in the loaded GUI. For example, the new QEGui customisation system uses this functionality to allow a custom menu item to be added that will interact with specific QE widgets in a dynamicly loaded GUI.
.
- Externalisation of QE widget controls.
    Applications can request QE widgets hand over user interface controls for hosting within some part of the application. For example, the QEImage widget can display recorder/playback controls as part of the widget. A QEGui application customisation may request that a QEImage widget in a GUI hand over these recorder controls to be loaded into a dock.
.
- QEGui customisation...
    Without customisation the QEGui application is a rich generic display manager. Customisation files can be applied to allow QEGui to function as a specialised application rather than just a display manager. For example, application specific menu bar menus and items can be added to interact directly with QE widgets loaded from .ui files, interact with widgets within GUIs and launch other applications.

=======================

Tab detatch

New widgets and classes:
========================
QEUtilities: 
 This class provides a number of general purpose utility/support functions used by a number of the QE Framework widgets.
 All the functions are static – this class maintains no state data. The header file (QECommon.h) also provides a number of utility macros.
 This class is exported from the framework as the functionality may be useful in the development of 3rd party widgets. 

QEStripChart/QEPlotter
 These widgets have been modified to use the QEGraphic class, which ensures a common look and feel, including standardised mouse and wheel zooming, animated axis re-scaling; andp rovides wrapper functions to hide QWT version API changes.

QEResizeableFrame. 
 Internally used widget that allows a frame edge to be selected and dragged in order to resize the frame. 
 It is not available as a designer widget.

QEArchiveStatus
 Provides status on the archives (traditional channel archivers) defined by the QE_ARCHIVE_LIST environment variable, namely hostname, port and endpoint as defined by QE_ARCHIVE_LIST together with status, number of indices available, number read and PV count.

QEArchiveNameSearch
 While the strip chart uses a strict name search using regular expressions, this widget provides a more relaxed means of searching for PV names available from the archiver. E.g. searching for “ABC  DEF” finds any PV name that contains ABC and/or DEF in any order, and is case insignificant. The usual right click context menu applies to each item in the list.

QEGeneralEdit
 When in engineer user level, and additional context menu appears – “Edit PV” which brings up a generic PV value editor.
 For numeric types, it presented a QENumericEdit widget, for enumeration types it presents a QERadioGroup, and for string types is presents a QELineEdit widget to the user.

QEDialog
  This is the base class for a number of dialogs.  It allows the dialog to be centred over the widget or  window of interest as opposed to the centre of the screen.

Enhanced Widgets and classes:
=============================
QEImage enhancements:
  - User control of image display properties (brightness/contrast,
    false colour, contrast reversal, logarithmic brightness)
  - Recording and playback
  - MPEG sources (as well as CA)
  - Direct access to QEImage widgets functions from custom QEGui menu items.
  - Allow external controls so applications like QEGui can appropriate
    controls. For example, QEGui can be customised to locate the recording
    controls as a dock.
  - Add processing for more image formats.
  - Improved user intraction with image.
  - Improved integration with area detector. When a user interacts with
    an image to select a profile through the image, the generation of the
    profile data can be performed within the QEImage widget, or by area
    detector.
  - Full screen mode

Deprecation:
============
QE Widgets deprecated in version 2.0 have been removed. All deprecated widgets were only unique in name only, being light wrappers around current QE widgets. For example, QCaLabel was implemented using QELabel. Any code or .ui files referencing deprecared QCa widgets can be modified to use the current version by renaming any widget names from QCa<name> to QE<name>

to hold published profile reference.
In QEForm, change uiFile property to use standard variable name and
substitution mechanism, with the filename taking the place of the
variable name.

substitutions (or in the case of the QEForm widget, ui file name and
macro substitutions) are available.

QEForm, ui files were loaded) when a variable name (ui file name)
property was set during loading, then again when the macro substitutions
property was set.
Also in QEForm, load sub forms after processing events for loading
higher level forms has completed. This means deeply nested forms and
forms with many sub forms hidden in tabs appear to load much quicker and
are interactive earlier.

are interactive earlier.

epicsqt 2.5.0 Release Notes

==================================================================================================
Changes between 2.5 and 2.4

SUMMARY:
Addition of new QE widgets: QENumericEdit, QECheckBox, QERadioGroup.
Expand documentation (reference and guide).
Reduce startup time for embedded sub forms.
Display various information in 'Help->About' including macro substitutions, search paths, open files, version numbers of application, libraries and plugins.
Add persistence management. Configurations such as window layout can be saved and restored at any time including startup.
Add QEGui application logo.
QEGui executable is now named qegui.
Allow multiple gui files to be opened on startup.
QEGui has display scaling capability (-a startup parameter)
Add application wide user level passwords. Implemented in QEGui.
If QELogin widget is used in an application that does not use this feature, the QELogin widget maintains its own local passwords as before.
QERadionButton and QECheckBox now defaults to write "1" when checked and "0" when unchecked.
QEGui path list now uses : or ; (depending on platform) to sperate paths so spaces can be used within paths.
Bug fixes. (See detailed log)

an application (example QEGui) and once when a QEForm is loaded by the
Qt .ui loader. On Linux this doesn't matter - the library is mapped once
and the same mapping is returned in both cases. So any static
variables only exist once. In windows, however, the library
(QEPlugin.dll) is mapped twice and static variables are created twice.
This can be observed by printing out the address of a static variable.
The 'same' static variable will have different addresses depening on
whether it is being referenced by the application or the Qt .ui form
loader.
Debug statement are still present.
